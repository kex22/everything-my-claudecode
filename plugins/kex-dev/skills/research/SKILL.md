# 技术调研

深入研究技术问题，产出调研报告供决策参考。

**开始时宣布：** "我正在使用 research 技能来调研这个问题。"

## 触发场景

- **用户主动调用**：直接发起调研
- **其他技能触发**：在设计、规划、执行过程中遇到不确定的技术问题时，暂停当前流程，先调研再继续

## 调研流程

### 1. 明确问题
- 精确定义要调研的问题（不是"了解 X"，而是"X 在 Y 场景下是否适用"）
- 确定调研范围和深度
- 如果由其他技能触发，记录调用上下文（哪个技能、什么阶段、什么问题）

### 2. 多源调研

按优先级使用以下信源：

| 信源 | 价值 | 适用场景 |
|------|------|----------|
| 官方文档 | 权威、准确 | API 用法、配置参数、版本变更 |
| Context7 | 最新文档 + 代码示例 | 快速获取库的最新用法 |
| GitHub Issues/Discussions | 真实 bug 和解决方案 | 排查已知问题、了解 roadmap |
| Reddit (r/rust, r/webdev 等) | 真实使用体验、踩坑记录 | 技术选型、方案对比、社区口碑 |
| Stack Overflow | 具体问题的解答 | 特定报错、用法疑问 |
| 技术博客 | 深度分析和实践经验 | 架构设计、性能优化 |

- 对比竞品/替代方案（至少 2-3 个）
- 如有必要，编写验证代码（POC）

### 3. 产出调研报告

保存到 `docs/research/<topic>.md`，格式：

```markdown
# <调研主题>

> 调研日期：YYYY-MM-DD
> 触发来源：用户主动 / design / plan / execute
> 关联设计：docs/design/<name>.md（如有）

## 问题
要解决什么问题，为什么需要调研

## 结论
先给结论，再展开分析

## 分析
对比各方案的优劣

## 参考
信息来源链接
```

### 4. 回归调用方
- 如果由其他技能触发，调研完成后回到原流程继续
- 如果调研结论影响已有设计，提示用户是否需要更新设计文档
